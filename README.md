# Mini NPU Simulator

## 1. 프로젝트 개요

이 프로젝트는 Python으로 만든 콘솔 기반 Mini NPU 시뮬레이터입니다. 사용자는 3x3 필터와 패턴을 직접 입력해서 MAC(Multiply-Accumulate) 연산 결과를 확인할 수 있고, `data.json`에 저장된 5x5, 13x13, 25x25 패턴을 한꺼번에 분석할 수도 있습니다. 프로그램은 입력 패턴과 필터의 위치별 곱셈 결과를 모두 더해 점수를 계산하고, 어느 필터와 더 유사한지 판정합니다.

이번 과제의 핵심은 단순히 “십자가냐 X냐”를 맞히는 데 있지 않습니다. 사람이 직관적으로 보던 모양 인식을, 숫자 배열과 반복문만으로 어떻게 흉내낼 수 있는지 직접 구현해보는 데 의미가 있습니다. 특히 MAC 연산, 라벨 정규화, 부동소수점 비교 정책, 크기 검증, 성능 측정까지 한 흐름으로 묶어서 다루는 점이 이 과제의 중심입니다.

## 2. 실행 환경

```bash
$ python3 --version
Python 3.8+
```

설명:

- 이 과제는 Python 3.8 이상 기준으로 작성했습니다.
- 외부 라이브러리 없이 표준 라이브러리(`json`, `time`)만 사용했습니다.
- NumPy 같은 벡터화 라이브러리는 사용하지 않고, 반복문으로 직접 MAC 연산을 구현했습니다.

## 3. 실행 방법

```bash
$ python3 main.py
```

프로그램을 실행하면 아래와 같은 메뉴가 표시됩니다.

```text
=== Mini NPU Simulator ===

[모드 선택]
1. 사용자 입력 (3x3)
2. data.json 분석
```

실행 방식:

- `1`: 사용자 입력 모드입니다. 3x3 필터 A, 필터 B, 패턴을 직접 입력하고 즉시 점수와 판정 결과를 확인합니다.
- `2`: `data.json` 분석 모드입니다. 파일에 저장된 필터와 패턴을 읽어서 일괄 판정, 성능 분석, 결과 요약까지 수행합니다.

## 4. 기능 목록

이 프로그램은 아래 기능을 제공합니다.

1. **사용자 입력 기반 3x3 판별**
   - 3x3 필터 두 개와 3x3 패턴을 콘솔에서 입력받습니다.
   - 각 필터에 대한 MAC 점수를 계산합니다.
   - 두 점수를 비교해 `A`, `B`, `UNDECIDED` 중 하나로 판정합니다.

2. **JSON 기반 일괄 분석**
   - `data.json`에서 `size_5`, `size_13`, `size_25` 필터를 로드합니다.
   - 각 패턴의 `key`에서 크기 정보를 추출해 알맞은 필터를 선택합니다.
   - 패턴별 점수, 판정, expected 대비 `PASS/FAIL` 결과를 출력합니다.

3. **라벨 정규화**
   - `+`, `cross`, `x`처럼 제각각 들어올 수 있는 라벨을 내부적으로 `Cross`, `X`로 표준화합니다.
   - 이 과정을 통해 출력과 비교 기준을 하나로 통일했습니다.

4. **입력 검증 및 오류 처리**
   - 숫자 파싱 실패, 열 개수 불일치 같은 입력 오류를 감지합니다.
   - JSON 분석 시 크기 불일치나 필터 데이터 문제는 프로그램 전체를 중단하지 않고 해당 케이스만 `FAIL` 처리합니다.

5. **성능 측정**
   - MAC 연산 시간을 ms 단위로 측정합니다.
   - 최소 10회 반복 측정 후 평균 시간을 출력합니다.
   - 패턴 크기에 따라 연산량이 어떻게 증가하는지 확인할 수 있습니다.

6. **결과 요약 리포트**
   - 전체 테스트 수, 통과 수, 실패 수를 출력합니다.
   - 실패 케이스가 있으면 어떤 이유로 실패했는지 함께 보여줍니다.

## 6. 주요 함수와 역할 분리

### `normalize_label()`

- 입력 라벨을 표준 라벨로 바꾸는 함수입니다.
- `cross`, `+`는 `Cross`로, `x`는 `X`로 변환합니다.
- 비교 기준을 통일하기 위한 전처리 역할을 합니다.

### `mac(pattern, filter_matrix)`

- 패턴과 필터의 같은 위치 값을 곱하고 모두 더하는 핵심 함수입니다.
- 이 프로그램에서 가장 중요한 연산이며, 외부 라이브러리 없이 반복문으로 직접 구현했습니다.

### `get_judge(score_a, score_b, label_a, label_b)`

- 두 점수를 비교해 어느 쪽이 더 큰지 판정합니다.
- 두 점수 차이가 `EPSILON`보다 작으면 `UNDECIDED`를 반환합니다.

### `input_matrix(name, size)`

- 사용자로부터 `size x size` 행렬을 입력받는 함수입니다.
- 숫자 파싱 오류, 열 개수 오류를 검사하며 재입력을 유도합니다.

### `measure_performance(pattern, filter_matrix, iterations=10)`

- MAC 연산 시간을 반복 측정해 평균 ms 값을 계산하는 함수입니다.
- 성능 분석 표의 기초 데이터를 담당합니다.

### `process_filters(raw_filters)`

- JSON 파일의 필터 데이터를 읽고, 크기별 필터 맵으로 정리하는 함수입니다.
- 필터 라벨 정규화도 함께 수행합니다.

### `analyze_pattern(pattern_item, processed_filters, performance_data)`

- JSON 패턴 하나를 분석하는 함수입니다.
- 키 검증, 크기 검증, 점수 계산, 판정, `PASS/FAIL` 판별을 한 번에 처리합니다.

### `mode_user_input()`

- 사용자 입력 모드 전체 흐름을 담당합니다.
- 필터 입력, 패턴 입력, 점수 계산, 판정, 성능 분석을 순서대로 실행합니다.

### `mode_json_analysis()`

- `data.json` 분석 모드 전체 흐름을 담당합니다.
- 필터 로드, 패턴 분석, 성능 요약, 결과 요약을 묶어서 처리합니다.

정리하면, `mac()`는 **연산 핵심**, `analyze_pattern()`은 **케이스 분석 핵심**, 각 모드 함수는 **실행 흐름 제어**를 맡고 있습니다.

## 7. 입력 처리와 예외 처리

이 프로그램은 입력과 데이터 검증을 꽤 중요하게 다룹니다. 이런 과제에서 이상한 입력 하나 들어왔다고 프로그램이 그대로 쓰러지면 좀 민망하니까요.

처리 기준은 아래와 같습니다.

- 사용자 입력 모드에서 각 줄은 공백 기준으로 분리합니다.
- 숫자로 변환할 수 없는 값이 들어오면 오류 메시지를 출력합니다.
- 각 줄의 숫자 개수가 지정된 크기와 다르면 재입력을 유도합니다.
- JSON 분석 모드에서 패턴 크기와 필터 크기가 맞지 않으면 해당 케이스만 `FAIL` 처리합니다.
- 지원하지 않는 크기의 필터가 없을 경우에도 전체 프로그램은 중단하지 않습니다.

예를 들어 아래 상황을 처리할 수 있습니다.

```text
1 0
abc def ghi
1 0 1 0
```

또한 `data.json`에 잘못된 키 형식이나 잘못된 행렬 크기가 들어와도, 전체 분석이 멈추는 대신 어떤 케이스가 왜 실패했는지 드러내도록 구성했습니다.

## 8. 데이터 구조와 JSON 형식

이 프로젝트는 `data.json` 파일을 사용합니다.

현재 데이터 구조는 아래와 같습니다.

```json
{
  "filters": {
    "size_5": {
      "cross": [[...]],
      "x": [[...]]
    },
    "size_13": {
      "cross": [[...]],
      "x": [[...]]
    },
    "size_25": {
      "cross": [[...]],
      "x": [[...]]
    }
  },
  "patterns": [
    {
      "key": "size_5_1",
      "input": [[...]],
      "expected": "x"
    }
  ]
}
```

필드 설명:

- `filters`: 크기별 필터 집합
- `size_5`, `size_13`, `size_25`: 각 크기에 대응하는 필터 그룹
- `cross`, `x`: 실제 필터 행렬 라벨
- `patterns`: 분석할 패턴 목록
- `key`: 패턴 크기와 번호 정보가 담긴 식별자
- `input`: 실제 입력 행렬
- `expected`: 기대 정답 라벨

동작 방식은 이렇습니다.

1. `filters`를 먼저 읽습니다.
2. 각 필터 라벨을 `Cross`, `X`로 정규화합니다.
3. `patterns`의 각 항목에서 `key`를 확인합니다.
4. `size_N_idx` 형식에서 `N`을 추출합니다.
5. 해당 크기의 필터를 가져와 MAC 연산을 수행합니다.
6. 판정 결과와 `expected`를 비교해 `PASS/FAIL`을 결정합니다.

## 9. 구현 포인트 요약

이 과제에서 중요했던 포인트는 아래 네 가지였습니다.

### 1) 라벨 정규화

입력 데이터는 `+`, `cross`, `x`처럼 제각각 들어올 수 있습니다. 이걸 그대로 비교하면, 사실 같은 의미인데 문자열이 다르다는 이유로 실패하는 황당한 일이 생깁니다. 그래서 내부에서는 무조건 `Cross`, `X` 두 값만 쓰도록 정리했습니다.

### 2) MAC 연산 직접 구현

이 과제는 일부러 NumPy를 못 쓰게 했기 때문에, 진짜로 반복문을 돌면서 같은 위치끼리 곱하고 더해야 합니다. 귀찮지만 이게 핵심입니다. 결국 AI 하드웨어가 왜 반복 연산에 강해야 하는지도 여기서 연결됩니다.

### 3) Epsilon 기반 동점 처리

부동소수점은 보기보다 꽤 음흉합니다. 출력만 보면 같아 보여도 실제 내부 값은 미세하게 다를 수 있습니다. 그래서 `abs(score_a - score_b) < 1e-9` 조건을 두고, 이 경우는 억지로 승자를 정하지 않고 `UNDECIDED`로 처리했습니다.

### 4) 케이스 단위 실패 처리

JSON 데이터에 오류가 있다고 해서 전체 프로그램이 멈추는 건 별로 좋은 태도가 아닙니다. 그래서 스키마 오류, 크기 불일치, 필터 문제는 해당 케이스만 `FAIL` 처리하고 다음 분석을 계속하도록 만들었습니다.

## 10. 결과 리포트

### 1. 실패 원인 분석

현재 `data.json` 테스트 결과는 **0 FAIL**입니다. 즉, 현재 포함된 모든 패턴이 기대값과 일치하는 판정을 받았습니다. 이 결과가 나온 이유는 운이 좋아서가 아니라, 비교 정책과 데이터 정리가 꽤 명확하게 들어갔기 때문입니다.

첫째, **라벨 정규화**가 제대로 들어가 있습니다. `expected` 값으로 들어오는 `+`는 `Cross`로, `x`는 `X`로 변환됩니다. 필터 쪽도 `cross`, `x`를 같은 기준으로 정리하기 때문에, 문자열 표기 차이 때문에 생길 수 있는 불필요한 실패를 막았습니다.

둘째, **부동소수점 비교 정책**을 따로 뒀습니다. 점수 차이가 거의 없는 상황에서는 미세한 float 오차 때문에 억지 승부가 나는 일이 생길 수 있습니다. 이를 막기 위해 `EPSILON = 1e-9`를 사용했고, 차이가 충분히 작으면 `UNDECIDED`로 처리했습니다. 덕분에 “사실상 같은 점수인데 한쪽이 이긴 걸로 찍히는” 이상한 결과를 줄일 수 있었습니다.

셋째, **크기 검증과 케이스 단위 실패 처리**를 넣었습니다. 패턴 키에서 추출한 크기와 실제 행렬 크기가 다르면 해당 케이스는 `FAIL`로 처리됩니다. 중요한 건 여기서 프로그램 전체가 죽지 않는다는 점입니다. 즉, 데이터 문제와 로직 문제를 분리해서 볼 수 있게 만들었습니다.

넷째, 현재 `data.json` 자체도 과제 요구에 맞게 정리되어 있습니다. `filters`는 크기별로 나뉘어 있고, `patterns`는 `key`, `input`, `expected`를 포함합니다. 다시 말해 입력 데이터와 코드가 같은 규칙을 보고 움직이고 있기 때문에 0 FAIL이라는 결과가 가능했습니다.

결론적으로, 지금 0 FAIL은 “그냥 다 맞았음” 수준이 아니라, **정규화 + epsilon 정책 + 크기 검증 + 안정적인 분석 흐름**이 맞물린 결과라고 볼 수 있습니다.

### 2. 시간 복잡도 및 성능 분석

성능 측정 결과를 보면, 패턴의 크기 `N`이 커질수록 MAC 연산 시간이 증가합니다. 이건 당연한데, 당연한 걸 코드와 측정으로 확인하는 게 과제의 포인트입니다.

| 크기 (NxN) | 연산 횟수 (N²) | 예상 시간 복잡도 |
| :--- | :--- | :--- |
| 3x3 | 9 | O(N²) |
| 5x5 | 25 | O(N²) |
| 13x13 | 169 | O(N²) |
| 25x25 | 625 | O(N²) |

분석 근거는 단순합니다. `mac()` 함수는 `N x N` 행렬의 모든 위치를 한 번씩 순회합니다. 각 위치마다 곱셈 1번, 누적 덧셈 1번이 일어나므로 전체 연산량은 행렬 크기의 제곱에 비례합니다. 그래서 시간 복잡도는 **O(N²)**입니다.

예를 들어 5x5는 25칸, 25x25는 625칸입니다. 한 변 길이는 5배 차이지만 실제 연산 횟수는 25배 차이입니다. 이게 바로 제곱 복잡도입니다. 실제 측정 시간도 이 증가 흐름을 따라가는 쪽으로 나타납니다. 물론 운영체제 스케줄링이나 측정 오차 때문에 완벽히 비례하진 않겠지만, 전체 경향은 분명합니다.

즉, 이 프로그램은 작은 패턴에서는 빠르게 동작하지만, 크기가 커질수록 순수 반복 연산 비용이 빠르게 늘어납니다. 바로 이런 이유 때문에 실제 AI 연산에서는 CPU 대신 GPU나 NPU처럼 대량의 반복 계산에 최적화된 하드웨어가 중요해집니다.

## 11. 마무리

이 프로젝트는 겉으로 보면 단순한 패턴 판별기처럼 보이지만, 실제로는 AI 연산의 아주 축소된 핵심을 직접 만져보는 과제였습니다. 숫자 배열, 반복문, 점수 비교, float 오차, 데이터 스키마, 성능 측정까지 다 한 번에 건드리게 되어 있어서 생각보다 배울 게 많았습니다.

특히 “모양을 본다”는 인간의 직관을, 컴퓨터는 결국 **행렬과 연산**으로만 다룬다는 점이 잘 드러납니다. 거창한 모델이 없어도, 필터와 MAC만으로도 패턴 인식의 아주 기초적인 형태를 구현할 수 있다는 걸 확인할 수 있었습니다.
